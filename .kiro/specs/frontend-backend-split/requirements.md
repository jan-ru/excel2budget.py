# Requirements Document

## Introduction

This specification covers splitting the current monolithic Python data conversion tool into a two-tier architecture: a FastAPI backend (Python) and a TypeScript frontend. The backend owns template definitions, documentation generation, customer configuration persistence (DuckDB), and the CLI. The frontend owns all data processing: Excel import, DuckDB-WASM SQL transformation, IronCalc-WASM spreadsheet rendering, validation, export (CSV/Excel/PDF), and the UI shell. Core types flow from Pydantic models on the backend through auto-generated OpenAPI to TypeScript types via openapi-typescript, establishing a single source of truth for the API contract.

## Glossary

- **Backend**: The FastAPI Python server responsible for template registry, documentation generation, customer/config persistence, and CLI tooling
- **Frontend**: The plain TypeScript browser application (no framework such as React, Vue, Angular, or full SAP UI5) responsible for data import, transformation (DuckDB-WASM), spreadsheet rendering (IronCalc-WASM), validation, export, and the UI shell; uses @ui5/webcomponents (SAP OpenUI5 standalone web components) for UI elements where useful
- **API_Gateway**: The FastAPI application that serves REST endpoints and auto-generates the OpenAPI schema from Pydantic models
- **Template_Registry**: The backend module that stores and serves accounting package output template definitions (afas, exact, twinfield)
- **Documentation_Module**: The backend module that generates all 7 documentation artifacts (ArchiMate, BPMN, input/output/transform descriptions, control table, user instruction)
- **Config_Store**: The DuckDB persistent file on the backend used for customer configuration and template metadata persistence
- **Type_Pipeline**: The toolchain that generates TypeScript types from Pydantic models: Pydantic → OpenAPI spec (auto-generated by FastAPI) → openapi-typescript → TypeScript types
- **Pipeline_Orchestrator**: The frontend module that coordinates the client-side data flow: import → validate → transform → preview → export
- **SQL_Generator**: The frontend module that generates DuckDB SQL for the unpivot + DC split transformation
- **DuckDB_WASM**: The WebAssembly build of DuckDB running in the browser for SQL transformation
- **IronCalc_WASM**: The WebAssembly build of IronCalc running in the browser for spreadsheet rendering
- **XSS_Sanitizer**: The frontend module that sanitizes cell values before browser rendering to prevent cross-site scripting
- **Excel_Importer**: The frontend module that parses .xlsx files client-side
- **CSV_Excel_Exporter**: The frontend module that exports transformed data as CSV or Excel downloads
- **PDF_Exporter**: The frontend module that generates PDF documents from screen content using jsPDF or pdf-lib
- **Memory_Guard**: The frontend module that validates file sizes and estimated memory usage against browser WASM limits
- **Data_Validator**: The frontend module that performs pre-transform validation on imported data
- **UI_App**: The frontend application shell managing screens and user interactions, built with plain TypeScript and @ui5/webcomponents
- **UI5_Web_Components**: The @ui5/webcomponents npm package — a lightweight, framework-agnostic set of SAP OpenUI5 web components that run standalone without the full SAP UI5 framework
- **ApplicationContext**: A metadata/aggregate object sent from the frontend to the backend for documentation generation; contains no raw customer financial data
- **OpenAPI_Spec**: The JSON/YAML schema auto-generated by FastAPI from Pydantic model definitions

## Requirements

### Requirement 1: Backend Project Structure

**User Story:** As a developer, I want the backend to be a standalone FastAPI Python project, so that it can be deployed, versioned, and tested independently from the frontend.

#### Acceptance Criteria

1. THE Backend SHALL be structured as a Python project with FastAPI as the web framework and Pydantic as the data modeling layer
2. THE Backend SHALL contain the following modules: Template_Registry, Documentation_Module, Config_Store, CLI, and API_Gateway
3. THE Backend SHALL define all shared domain types as Pydantic models in a dedicated core types module
4. THE API_Gateway SHALL auto-generate an OpenAPI_Spec from the Pydantic model definitions at startup
5. WHEN the Backend starts, THE API_Gateway SHALL serve the OpenAPI_Spec at a well-known endpoint (e.g. `/openapi.json`)

### Requirement 2: Frontend Project Structure

**User Story:** As a developer, I want the frontend to be a standalone TypeScript project, so that it can be developed, built, and tested independently from the backend.

#### Acceptance Criteria

1. THE Frontend SHALL be structured as a plain TypeScript project with a bundler (e.g. Vite); the Frontend SHALL NOT use a framework such as React, Vue, Angular, or the full SAP UI5 framework
2. THE Frontend SHALL use @ui5/webcomponents (SAP OpenUI5 standalone web components) for UI elements where useful; the Frontend SHALL NOT depend on the full SAP UI5 framework
3. THE Frontend SHALL contain the following modules: Excel_Importer, SQL_Generator, Pipeline_Orchestrator, DuckDB_WASM engine, IronCalc_WASM engine, XSS_Sanitizer, Data_Validator, CSV_Excel_Exporter, PDF_Exporter, Memory_Guard, and UI_App
4. THE Frontend SHALL be fully ephemeral: no user data or financial data SHALL be persisted to disk, local storage, or any remote server
5. WHEN the browser session ends, THE Frontend SHALL release all in-memory data without writing to persistent storage

### Requirement 3: Type Synchronization Pipeline

**User Story:** As a developer, I want TypeScript types to be auto-generated from the backend Pydantic models, so that the API contract between frontend and backend stays in sync without manual duplication.

#### Acceptance Criteria

1. THE Type_Pipeline SHALL generate TypeScript type definitions from the OpenAPI_Spec produced by the API_Gateway
2. THE Type_Pipeline SHALL use openapi-typescript (or equivalent) to convert the OpenAPI_Spec into TypeScript types
3. WHEN a Pydantic model changes on the Backend, THE Type_Pipeline SHALL regenerate the corresponding TypeScript types to reflect the change
4. THE Type_Pipeline SHALL be executable as a single CLI command (e.g. `npm run generate-types`)
5. THE generated TypeScript types SHALL be the sole source of API type definitions used by the Frontend; the Frontend SHALL NOT define duplicate API types manually

### Requirement 4: Template Registry API

**User Story:** As a frontend developer, I want to fetch available accounting packages and template definitions from the backend, so that the frontend can present template selection without bundling template logic.

#### Acceptance Criteria

1. THE API_Gateway SHALL expose a GET endpoint that returns the list of available accounting package names
2. THE API_Gateway SHALL expose a GET endpoint that returns the list of available template names for a given package
3. THE API_Gateway SHALL expose a GET endpoint that returns the full OutputTemplate definition (column definitions with source mappings) for a given package and template combination
4. IF a requested package or template does not exist, THEN THE API_Gateway SHALL return an error response with available packages or templates listed
5. THE Template_Registry SHALL continue to support the existing template definitions: afas, exact, and twinfield

### Requirement 5: Documentation Generation API

**User Story:** As a user, I want to generate documentation artifacts from the backend after a transformation, so that documentation generation logic stays server-side while raw financial data stays client-side.

#### Acceptance Criteria

1. THE API_Gateway SHALL expose a POST endpoint that accepts an ApplicationContext and returns a DocumentationPack containing all 7 artifacts
2. THE ApplicationContext sent by the Frontend SHALL contain only metadata and aggregates (system descriptors, process steps, column descriptions, control totals, transform descriptor), not raw customer financial data
3. THE Documentation_Module SHALL generate all 7 artifacts: ArchiMate diagram, BPMN diagram, input description, output description, transform description, control table, and user instruction
4. IF the ApplicationContext is incomplete or invalid, THEN THE API_Gateway SHALL return a descriptive error response

### Requirement 6: Customer Configuration Persistence

**User Story:** As a user, I want my customer-specific configurations (selected package, template, budgetcode, year) to be saved on the backend, so that I can reuse them across sessions.

#### Acceptance Criteria

1. THE Config_Store SHALL use a DuckDB persistent file for storing customer configuration data
2. THE API_Gateway SHALL expose endpoints to create, read, update, and list customer configurations
3. THE Config_Store SHALL store at minimum: configuration name, package name, template name, budgetcode, year, and creation/modification timestamps
4. WHEN a configuration is requested by name, THE API_Gateway SHALL return the full configuration or an error if not found
5. THE Backend SHALL NOT use SQLite or any additional database dependency beyond DuckDB for configuration persistence

### Requirement 7: Frontend Excel Import

**User Story:** As a user, I want to upload and parse Excel budget files entirely in the browser, so that my financial data never leaves my machine.

#### Acceptance Criteria

1. THE Excel_Importer SHALL parse .xlsx files client-side in the browser without transmitting file contents to any server
2. THE Excel_Importer SHALL extract budget data from the "Budget" sheet, including column headers and typed cell values
3. THE Excel_Importer SHALL extract the column mapping configuration (Entity, Account, DC, and Dutch month columns) from the sheet headers
4. IF the uploaded file is not a valid .xlsx or lacks the expected sheet, THEN THE Excel_Importer SHALL return a descriptive error to the user

### Requirement 8: Frontend SQL Generation and Transformation

**User Story:** As a user, I want the budget data transformation (unpivot + DC split) to run entirely in the browser using DuckDB-WASM, so that no financial data is sent to the backend.

#### Acceptance Criteria

1. THE SQL_Generator SHALL generate DuckDB-compatible SQL for the unpivot and DC split transformation based on the MappingConfig, OutputTemplate, and UserParams
2. THE SQL_Generator SHALL produce SELECT-only queries; the SQL_Generator SHALL NOT generate DDL or DML statements
3. THE DuckDB_WASM engine SHALL execute the generated SQL in the browser and return the transformed data
4. THE SQL_Generator SHALL use quoted identifiers to prevent SQL injection
5. IF SQL generation or execution fails, THEN THE Pipeline_Orchestrator SHALL return a descriptive error to the user

### Requirement 9: Frontend Data Validation

**User Story:** As a user, I want my imported data to be validated before transformation, so that I get clear error messages for invalid inputs.

#### Acceptance Criteria

1. THE Data_Validator SHALL validate that the MappingConfig references columns that exist in the imported data
2. THE Data_Validator SHALL validate that UserParams contain a non-empty budgetcode and a positive year
3. THE Data_Validator SHALL detect and report invalid DC values (values other than "D" or "C") with row-level detail
4. THE Data_Validator SHALL validate that month column period numbers are unique and in the range 1 to 12

### Requirement 10: Frontend Spreadsheet Rendering

**User Story:** As a user, I want to preview imported and transformed data in a spreadsheet view in the browser, so that I can visually verify the data before exporting.

#### Acceptance Criteria

1. THE IronCalc_WASM engine SHALL render tabular data in a spreadsheet view within the browser
2. THE XSS_Sanitizer SHALL sanitize all cell values before rendering to prevent cross-site scripting attacks
3. THE XSS_Sanitizer SHALL strip HTML tags, script elements, event handler attributes, and dangerous URI schemes from cell values

### Requirement 11: Frontend Export (CSV, Excel, PDF)

**User Story:** As a user, I want to export transformed data as CSV, Excel, or PDF directly from the browser, so that I can download results without server involvement.

#### Acceptance Criteria

1. THE CSV_Excel_Exporter SHALL export transformed TabularData as a CSV file download in the browser
2. THE CSV_Excel_Exporter SHALL export transformed TabularData as an Excel (.xlsx) file download in the browser
3. THE PDF_Exporter SHALL generate PDF documents from screen content (spreadsheet, diagram, control table) using a client-side PDF library (jsPDF or pdf-lib)
4. THE PDF_Exporter SHALL include metadata in the PDF: screen title, configuration name, accounting package, template name, and generation timestamp
5. WHEN the user triggers an export, THE Frontend SHALL NOT transmit the exported data to any server

### Requirement 12: Frontend Memory and WASM Guards

**User Story:** As a user, I want the application to prevent me from uploading files that would exceed browser memory limits, so that the browser tab does not crash.

#### Acceptance Criteria

1. THE Memory_Guard SHALL validate file size against a configurable maximum before parsing
2. THE Memory_Guard SHALL estimate the in-memory footprint (file size × expansion factor) and compare it against the WASM memory limit
3. IF the file size exceeds the maximum or the estimated memory exceeds the WASM limit, THEN THE Memory_Guard SHALL reject the file with a descriptive error message before parsing begins

### Requirement 13: Frontend Pipeline Orchestration

**User Story:** As a user, I want the frontend to coordinate the full conversion flow (import → validate → transform → preview → export), so that I experience a seamless workflow.

#### Acceptance Criteria

1. THE Pipeline_Orchestrator SHALL coordinate the following steps in order: file import, data validation, template retrieval (from Backend API), SQL generation, DuckDB-WASM execution, result preview, and export
2. THE Pipeline_Orchestrator SHALL fetch the OutputTemplate from the Backend API before SQL generation
3. IF any step in the pipeline fails, THEN THE Pipeline_Orchestrator SHALL halt the pipeline and present the error to the user without proceeding to subsequent steps
4. THE Pipeline_Orchestrator SHALL operate entirely in-memory; no intermediate data SHALL be persisted

### Requirement 14: Frontend UI Application Shell

**User Story:** As a user, I want a multi-screen UI that guides me through the conversion workflow, so that I can upload, configure, transform, preview, export, and view documentation.

#### Acceptance Criteria

1. THE UI_App SHALL present the following screens using plain TypeScript and UI5_Web_Components: Upload, Preview, Configuration (template selection + parameter input), Transform, Output, and Documentation
2. WHEN the user is on any screen, THE UI_App SHALL display the current date (YYYY-MM-DD) and offer a "Download as PDF" action
3. THE UI_App SHALL fetch the list of available packages and templates from the Backend API for the Configuration screen
4. WHEN the user triggers documentation generation, THE UI_App SHALL build an ApplicationContext from the current session data and send it to the Backend documentation endpoint

### Requirement 15: Backend CLI Preservation

**User Story:** As a developer or ops engineer, I want the CLI to remain on the backend as a dev/ops tool, so that I can run conversions from the command line without the frontend.

#### Acceptance Criteria

1. THE CLI SHALL remain in the Backend project and continue to support the existing command-line interface: positional arguments (input_file, package, template), required flags (--budgetcode, --year), and optional flags (-o, -f, -v, -q, --list-packages, --list-templates, --version)
2. THE CLI SHALL use the same Pydantic-based core types and Template_Registry as the API_Gateway
3. THE CLI SHALL exit with code 0 on success, 1 on input/configuration errors, and 2 on transformation/export errors

### Requirement 16: Backend DuckDB Dual Usage

**User Story:** As a developer, I want DuckDB to be used on both sides (native Python on backend for persistence, WASM in frontend for transformation), so that we avoid adding SQLite as an extra dependency.

#### Acceptance Criteria

1. THE Config_Store SHALL use DuckDB (native Python driver) for persistent storage of customer configurations and template metadata
2. THE Frontend SHALL use DuckDB-WASM for in-browser SQL transformation of budget data
3. THE Backend SHALL NOT use SQLite or any other relational database for persistence

### Requirement 17: API Contract and Error Handling

**User Story:** As a frontend developer, I want consistent, well-typed API responses from the backend, so that I can handle success and error cases uniformly.

#### Acceptance Criteria

1. THE API_Gateway SHALL return JSON responses with consistent structure for both success and error cases
2. THE API_Gateway SHALL use appropriate HTTP status codes: 200 for success, 400 for client errors, 404 for not found, 422 for validation errors, 500 for server errors
3. THE API_Gateway SHALL include descriptive error messages in error responses
4. THE OpenAPI_Spec SHALL document all request and response schemas, including error response schemas

### Requirement 18: Data Privacy Boundary

**User Story:** As a user, I want assurance that my raw financial data never leaves the browser, so that I can trust the tool with sensitive budget information.

#### Acceptance Criteria

1. THE Frontend SHALL process all raw financial data (Excel cell values, transformed rows, exported files) exclusively in the browser
2. THE Frontend SHALL NOT transmit raw financial data to the Backend or any external server at any point in the pipeline
3. THE ApplicationContext sent to the Backend for documentation generation SHALL contain only metadata and aggregates (system descriptors, column descriptions, control totals, process steps), not individual row data or cell values
4. THE Backend SHALL NOT receive, store, or process raw customer financial data
